## Objective

Implement the Azure CI/CD and release-orchestration layer for staging and controlled-pilot deployment without deploying production automatically, changing DNS, committing Azure credentials, creating customer data, or bypassing GitHub environment approvals.

## Build and Promotion Model

Staging builds API/web images once, tags them with the Git SHA, pushes to ACR, resolves digests, and records those digests in `artifacts/deployment-release/manifest.json`.

Production-pilot promotion consumes the staging deployment artifact and promotes the same recorded image refs and widget release artifacts. It does not rebuild for pilot.

Current caveat:

- Widget artifacts are environment-origin sensitive. Staging artifacts must not be promoted to pilot if their embedded public origins differ from pilot origins. The deployment manifest records this as an explicit promotion note until environment runtime resolution is implemented or separate environment artifacts are generated safely.

## Deployment Manifest

Generated by:

```bash
npm run azure:release:manifest -- --environment staging
```

Output:

- `artifacts/deployment-release/manifest.json`
- `artifacts/azure-deployment/<environment>/report.json`

The manifest records:

- Git SHA
- API image ref/digest
- web image ref/digest
- SDK version and checksums
- iframe checksum
- protocol major
- public API version
- Alembic head
- admin/pilot readiness report status
- promotion notes

No secrets are stored.

## Static Artifact Publication

Generated widget artifacts are published in safe order:

1. immutable SDK semantic loader
2. iframe HTML target content
3. hashed iframe assets
4. SDK major alias loader
5. alias metadata
6. release manifest

Immutable paths are never overwritten with different checksums. Same checksum no-op is allowed.

## Migration Execution

`npm run azure:migrate` updates the B1 migration job image and starts the job when `--execute` is provided. Dry-run is default.

Migrations are not executed by every API replica.

## Rollback

`npm run azure:rollback:plan` validates deployment-manifest compatibility and blocks automatic rollback if:

- protocol major differs
- public API version differs
- database migration head differs

Rollback execution is workflow-gated and does not downgrade databases automatically.
