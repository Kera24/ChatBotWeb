# Widget Origins, Public Key, and Embed Management

TASK-067B2 implements backend-only administration APIs for widget domains, public key rotation, and embed metadata. It does not add an admin frontend, preview grants, installation crawling, production deployment, or monitoring integration.

## Origin Model

Allowed origins continue to use the existing `CredentialAllowedOrigin` model attached to the widget's active `PublicCredential`. Origins belong to the stable widget/public-credential boundary, not to configuration revisions. This means origin changes can affect public runtime authorization immediately while historical published configuration revisions remain immutable.

Implemented routes:

- `GET /api/v1/workspaces/{workspace_id}/widgets/{widget_id}/origins`
- `POST /api/v1/workspaces/{workspace_id}/widgets/{widget_id}/origins`
- `DELETE /api/v1/workspaces/{workspace_id}/widgets/{widget_id}/origins/{origin_id}`

The service enforces tenant-scoped widget loading before origin access. It does not trust tenant or workspace values from the request body.

## Normalization And Validation

Origin administration reuses the existing credential origin validator and adds widget-specific exact-origin guardrails.

Rules:

- Exact origins only.
- Wildcards and `*` are rejected for widget administration.
- Scheme and host are lowercased.
- Default ports are removed: `https://example.com:443/` becomes `https://example.com`; `http://localhost:80/` becomes `http://localhost`.
- Paths other than `/`, query strings, fragments, and userinfo are rejected by the shared validator.
- Localhost and loopback origins remain development/test-only according to credential environment.
- Production credentials require HTTPS and reject local origins.
- Active origins are capped at 20 per widget.

Duplicate canonical origins are rejected through the existing normalized uniqueness constraint.

## Origin Removal Invariant

A published, operationally enabled widget must retain at least one active origin. The API rejects deletion of the final active origin while an active published revision exists and the widget remains enabled. Operators can add a replacement origin first or disable/unpublish through the appropriate workflow.

## Public Key Lifecycle

Widget public keys are public identifiers, not dashboard secrets. They identify a public widget configuration and are still protected by allowed-origin validation, operational/pilot policy, rate limits, session validation, and tenant-scoped retrieval.

Rotation is immediate cutover:

1. The caller supplies the expected current public credential ID.
2. The service tenant-scopes the widget and validates the expected credential.
3. The old credential is marked `revoked` with `rotated_at` and `revoked_at`.
4. A new unique public identifier is generated by the existing credential generator.
5. Active origins are copied to the replacement credential.
6. The stable `Widget.public_credential_id` points to the new credential.
7. Audit records key fingerprints only.

Old snippets stop resolving. The new key resolves the same active published widget revision. Configuration revisions, pilot status, operational status, and release channel are not mutated by rotation.

## Session After Rotation

Existing public sessions remain governed by the public session validation policy. Because rotation revokes the old credential and public routes validate credential status, old-key public flows are rejected instead of being silently rebound to the new key. The implementation does not add a dual-key grace period.

## Public Configuration Cache Behavior

Public configuration ETags now include a public-key cache key in addition to the sanitized public payload. This prevents a conditional request for the old key from creating a cross-key `304` response after rotation, even when the active configuration content is otherwise identical.

`Vary: Origin` remains required for public configuration responses.

## Embed Version Resolver

Approved SDK versions come from `deployment/widget/sdk-versions.json`, with optional enrichment from the generated B1 release manifest when present. The checked-in registry is provider-neutral and currently approves the foundation SDK release:

- SDK version: `0.1.0-foundation.0`
- SDK major: `1`
- protocol major: `1`
- public API version: `v1`
- managed alias path: `/widget-sdk/v1/loader.js`
- pinned immutable path: `/widget-sdk/v0.1.0-foundation.0/loader.js`

Unsupported, revoked, arbitrary URL, and `latest` selections are rejected.

## Managed And Pinned Embed Modes

`Widget` stores embed delivery preference outside configuration revisions:

- `embed_version_mode`: `managed_major` or `pinned`
- `pinned_sdk_version`: nullable semver when pinned

Existing widgets default to `managed_major` through migration `0011_widget_embed_preferences`.

`managed_major` uses the SDK major alias and intentionally has no fixed SRI. `pinned` uses the immutable semantic loader path and includes SRI only when the generated release manifest provides it.

## Embed Snippet

The embed service generates a deterministic script snippet from approved release metadata and the current public key:

```html
<script async src="https://cdn.example.com/widget-sdk/v1/loader.js" data-widget-key="PUBLIC_KEY"></script>
```

Pinned snippets may include:

```html
integrity="sha384-..." crossorigin="anonymous"
```

The snippet contains no session token, tenant ID, API secret, arbitrary API URL override, iframe origin override, inline bootstrap code, or unsupported `latest` path. Values are HTML-escaped before rendering.

The CDN origin is currently the provider-neutral placeholder used by release documentation. A later deployment integration can source this from production deployment configuration once actual domains are approved.

## Embed Metadata

`GET /embed` returns admin-only metadata:

- public key and credential status
- publication status
- operational status
- pilot status
- release channel
- version mode and selected SDK version/path
- protocol major and API version
- optional SRI
- current snippet
- active allowed origins
- active published revision reference
- readiness codes
- `embed_update_required` after rotation

The API does not expose session tokens, signing secrets, provider credentials, kill-switch internals, or storage paths.

## RBAC And Tenant Isolation

The B2 endpoints reuse the B1 authenticated widget administration dependency requiring `org_owner` or `client_admin`. Every operation resolves the widget by organisation, workspace, and widget ID. Cross-tenant origin, key, and embed operations return the existing safe not-found behavior.

Future tasks may split granular permissions such as `widget:origins_manage`, `widget:key_rotate`, and `widget:embed_view`; B2 stays aligned with the current RBAC implementation.

## Audit

TASK-067B2 records:

- `widget_origin_added`
- `widget_origin_removed`
- `widget_public_key_rotated`
- `widget_embed_version_changed`

Audit metadata includes bounded canonical origins, version preferences, and key fingerprints. It does not include session tokens, authorization headers, credential secrets, messages, answers, citation quotes, or full public keys for rotation events.

## Migration

Migration `0011_widget_embed_preferences` adds stable embed preference fields to `widgets`:

- `embed_version_mode`, default `managed_major`
- `pinned_sdk_version`, nullable

The migration does not change public runtime behavior for existing widgets.

## TASK-067B3 Frontend Implementation Note

The B2 origin/key/embed APIs are now surfaced through authenticated dashboard UI. Tenant admins can manage exact allowed origins, view/copy the current public key, rotate it through a confirmation dialog, select managed-major or approved pinned SDK embed mode, and copy the inert server-generated snippet. Pilot enablement and global operational controls remain separate.

## TASK-067B4 Update

The Embed page now displays passive installation evidence by allowed origin. Evidence is recorded only after a valid public configuration request from an approved origin and stores no session token, message body, answer, visitor identity, or conversation content.
